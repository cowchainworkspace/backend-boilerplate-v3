# Правила участі в Open Source Month

### 1. Загальні положення
- Мета ініціативи — створення загального темплейта для бекенда.
- Участь є добровільною, але активні учасники зможуть отримати призи.
- Переможці визначаються за кількістю змержених pull request (PR) та якістю їх реалізації.

### 2. Робота з GitHub Issues
- Брати в роботу issue можна, якщо воно має лейбл open-source-month.
- Якщо issue не має цього лейбла, його потрібно узгодити з тімлідом або автором репозиторію.
- Можна пропонувати власні ідеї у вигляді нових issue, але вони мають бути підтверджені хоча б одним тімлідом та отримати лейбл open-source-month.

### 3. Pull Requests (PR) та їх мерджинг
- Кожен PR має проходити код-рев’ю двох тімлідів перед мерджем.
- PR має відповідати стандартам кодування компанії та містити детальний опис змін.
- Якщо PR містить нову функціональність, він має супроводжуватися тестами (за можливості).
- PR може бути відхилений або повернутий на доопрацювання, якщо він не відповідає вимогам.
- Технічний борг та рефакторинг також допускаються, але мають бути оформлені як окремі issue.

### 4. Оцінювання та винагороди
- За кожен змержений PR учасник отримує 1 бал.
- Якщо PR вирішує складне або важливе issue, то оцінка може бути збільшена (за рішенням лідів).
- Переможці будуть обрані за сумарною кількістю балів після завершення Open Source Month.
- Топ-3 учасники з найбільшою кількістю балів отримають призи

### 5. Додаткові умови
- Не допускається спам PR, тобто мінімальні зміни без значного внеску.
- Учасники можуть обговорювати issue, об’єднуватися в команди та допомагати одне одному.

### 6. Робота над одним issue кількома учасниками
#### 1. Резервація issue
 - Issue бере в роботу той, хто першим прокоментує його на GitHub словами “Я беру це в роботу”.
 - Після цього тімлід додає лейбл in-progress і призначає цього розробника відповідальним.
 #### 2. Командна робота
 - Якщо над issue хочуть працювати двоє або більше людей, вони можуть домовитися між собою та працювати разом.
 - У такому випадку вони самі вирішують, як розподілити роботу та поділяють бали між собою (наприклад, 50/50 або за домовленістю).
 - В PR має бути вказано, хто саме вніс який внесок (наприклад, у description).
 #### 3. Конфлікт, якщо двоє людей зробили один і той самий PR
 - Якщо два розробники зробили PR на одне й те ж issue без координації, мерджиться той PR, який:
 - має кращу якість коду;
 - відповідає вимогам issue;
 - першим отримав 2 approve від тімлідів.
 - Якщо два PR рівноцінні, тімліди можуть запропонувати авторам об’єднати їх або врахувати роботу обох (за домовленістю).
